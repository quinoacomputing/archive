51  // SDAG code follows. See http://charm.cs.illinois.edu/manuals/html/
52  // charm++/manual.html, Sec. "Structured Control Flow: Structured Dagger".
53 
54  // High-level overview of the dependency and asynchronous call structure
55  // ---------------------------------------------------------------------
56  //
57  // Directed Acyclic Graph (DAG): DAG legend:
58  // ----------------------------- Own - Owned nodes reordered
59  // Req - Node IDs requested
60  // Own -- Pre -- Ord -- Low --- Cre Pre - Start preparing nodes IDs
61  // / | \ / | Ord - Nodes reordered
62  // Req - | -- Upp --- / Low - Lower bound received
63  // \ / Upp - Upper bound computed
64  // -- Par ---- Cre - Create workers
65  // Par - Partitioners participated
66  //
67  // Interpretation of the above DAG
68  // -------------------------------
69  // Control flow is from left to right and top to bottom. As the above
70  // graph shows, we wait for both Own and Req conditions to be complete
71  // before we start preparing node IDs for the requestor.
72  //
73  // Reordering of node IDs consists of two main parts: (1) Reordering those
74  // nodes that are unique on our PE, i.e., do not exist on any other PE.
75  // These nodes can be assigned a new node ID without any communication.
76  // (2) Reordering those nodes that connect to mesh cells on other PEs.
77  // Assignment of new node IDs, i.e., reordering, must be handled in a way
78  // that the same new node IDs will be assigned to the nodes that are
79  // physically the same but happen to exist on other PEs as well as on
80  // ours.
81  //
82  // Accordingly, reordering starts with the owned, i.e., independently
83  // reordered nodes, on each PE. When that is done, Own is triggered,
84  // signaling that the new order for the owned nodes are ready to be
85  // queried. On another path, every PE immediately sends out requests to
86  // fellow PEs for nodes that they cannot reorder by themselves. However,
87  // those can only be sent back when the PE they are requested from has
88  // finished reordering them, so we wait until Own is satisfied on that PE.
89  // Req is triggered when a PE has received a request. Only when Own and
90  // Req are both satisfied do we need to prepare a set of new node IDs for
91  // a requestor, hence the above logic in the SDAG. Since there can be
92  // multiple requests arriving before any (or some or all) of them can be
93  // satisfied, the requests are queued into a vector and when the runtime
94  // system calls prepare(), we fullfill all requests, clear the queue, and
95  // automatically re-enable the trigger for Own, i.e., wait only for Req,
96  // i.e., new requests.
97  //
98  // The second part of the graph, only happend after reordering is done,
99  // after the mesh reordering has been completed and started by a call to
100  // bounds() inside reordered(). Ord is a partial synchronization point,
101  // because it happens at different times on different PEs.
102 
103  // First the upper bounds of the node IDs are computed on each PE, and
104  // once that is done the lower bounds are communicated among the fellow
105  // Partitioner branches, since the lower bound of PE n is the upper bound
106  // of PE n-1. Only when both upper and lower bounds have been stored on a
107  // PE, can Worker chare array elements be inserted, done in create().
108  // Initiation of estimating the average and standard deviation of the
109  // communication cost of merging the linear system is done in the
110  // beginning of create(), which starts two reductions, which must be done
111  // after each other: the first one accumulates the sum of the cost from
112  // each PE, and once the average is available on Transporter, a second
113  // round accumulates the variances back to Transporter.
114  //
115  // While the tasks carried out as described in the previous paragraph lead
116  // to correct results, there is an additional trigger, _participated_,
117  // that needs to be enabled before create() is called. The role of
118  // participated_complete() is to make sure that all PEs participate in the
119  // distribution of new (reordered) mesh node IDs during reordering. In
120  // particular, since PE 0 has nothing to receive from others, if this
121  // trigger is not in place, PE 0 goes ahead of every other PE and calls
122  // create() which calls Worker constructors, a potentially heavy function.
123  // This is undesired at this point since PE 0 will surely need to answer
124  // to requests from other PEs with new node IDs, which can only be done
125  // (if the participated trigger is not in place) when the Workers have
126  // been created. This introduces unnecessarly delays and load imbalance
127  // during reordering. Thus requiring participated_complete() in addition
128  // to lower and upper, delays creating of Workers. As a result, PE 0 is
129  // ready for requests from other PEs for new node IDs earlier than without
130  // the participate trigger, which yields a lot tighter CPU utilization
131  // during reordering. The participated trigger is enabled in two places:
132  // (1) in prepare(), i.e., if a PE had to prepare some of its new node IDs
133  // for others, and (2) in neworder(), i.e., if a PE had to receive new
134  // node IDs from others. Thus node reordering is only considered complete
135  // if every PE has participated in communication of the new node IDs.
136  //
137  // Note that Pre, Ord, and Cre are partial synchronization points: while
138  // Pre always happens before Ord, and Cre on the same PE, they happen at
139  // different times on different PEs, so these are not considered global
140  // synchronization points.
