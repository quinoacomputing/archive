2 /*!
3  \file src/Control/SystemComponents.h
4  \copyright 2012-2015, J. Bakosi, 2016-2017, Los Alamos National Security, LLC.
5  \brief Operations on numbers of scalar components of systems of equations
6  \details Operations on numbers of scalar components of systems of equations,
7  e.g. multiple equation sets of a physical model or a set of (stochastic
8  ordinary or partial differential) equations of different kinds.
9 
10  _Problem:_ We are given a type that is a tk::tuple::tagged_tuple that
11  contains an arbitrary number of std::vectors of integers. The number of
12  vectors are fixed at compile-time (accessed via tags) but their (differing)
13  length is only known at run-time (after parsing user input). What we need
14  are functions that operate on this data structure and return, e.g., the
15  total number of components in the whole system or the offset in the whole
16  data structure for a given tag. The functions should thus be able to operate
17  on a list of types, i.e., a double for loop over all tags and associated
18  vectors - one at compile-time and the other one at run-time.
19 
20  _Example:_ An example is to define storage for systems of stochastic
21  differential equations as in walker::ctr::ncomps. In
22  [walker](walker.html) various types of stochastic differential
23  equations are implemented and numerically integrated in time in order to
24  compute and learn about their statistical behavior. Since different types of
25  systems can be parameterized differently, there can be several systems of
26  the same type (the number is user-defined so only known at run-time). For
27  example, it is possible to numerically integrate two systems of Dirichlet
28  equations, see DiffEq/Dirichlet.h, one with 4, the other one with 5 scalar
29  variables, parameterized differently, and estimate their arbitrary coupled
30  statistics and joint PDFs. This requires that each type of system
31  has a vector of integers storing the number of scalar variables.
32 
33  _Solution:_ Looping through elements of a tuple is done via [Boost]
34  (http://www.boost.org)'s [MetaProgramming Library (MPL)]
35  (http://www.boost.org/doc/libs/release/libs/mpl)'s [for_each]
36  (http://www.boost.org/doc/libs/release/libs/mpl/doc/refmanual/for-each.html).
37  Such operations on types happen at compile-time, i.e., the code runs inside
38  the compiler and only its result gets compiled into code to be run at
39  run-time. Advantages are abstraction and generic code that is independent
40  of the size and order of the tags in the tuple (and the associated vectors).
41  Though it is not of primary concern for this data structure, this solution
42  also generates efficient code, since part of the algorithm (for computing
43  the offset, the total number of components, etc.) runs inside the compiler.
44  All of this is kept generic, so it does not need to be changed when a new
45  pair of tag and system of equations are added to the underlying tuple.
46  Thus, _the main advantage is that changing the order of the vectors in the
47  tuple or adding new ones at arbitrary locations will not require change to
48  client-code._
49 
50  _An alternative approach:_ Of course this could have been simply done with a
51  purely run-time vector of vector of integers. However, that would not have
52  allowed a tag-based access to the different systems of equations. (Sure, one
53  can define an enum for equation indexing, but that just adds code to what is
54  already need to be changed if a change happens to the underlying data
55  structure.) As a consequence, a component in a vector of vector would have
56  had to be accessed as something like, ncomps[2][3], which is more
57  error-prone to read than ncomps< tag::dirichlet >( 3 ). Furthermore, that
58  design would have resulted in double-loops at run-time, instead of letting
59  the compiler loop over the types at compile-time and do the run-time loops
60  only for what is only known at run-time. Additionally, and most importantly,
61  _any time a new system is introduced (or the order is changed), all
62  client-code would have to be changed._
63 
64  For example client-code, see walker::Dirichlet::Dirichlet() in
65  DiffEq/Dirichlet.h, or walker::Integrator::Integrator in Walker/Integrator.h.
66 */

87 //! \brief Map associating a dependent variable of a differential equation to
88 //! its offset in its data array storing its dependent variables
89 //! \details We use a case-insensitive character comparison functor for the
90 //! offset map, since the keys (dependent variables) in the offset map are
91 //! only used to indicate the equation's dependent variable, however, queries
92 //! (using std::map's member function, find) can be fired up for both ordinary
93 //! and central moments (which are denoted by upper and lower case,
94 //! characters, respectively) for which the offset (for the same dependent
95 //! variable) should be the same.
96 using OffsetMap = std::map< char, ncomp_type, CaseInsensitiveCharLess >;

98 //! \brief Number of components storage
99 //! \details All this trickery with boost::mpl allows the code below to be
100 //! generic. As a result, adding a new component requires adding a single line
101 //! (a tag and its type) to the already existing list, see typedefs 'ncomps'.
102 //! The member functions, doing initialization, computing the number of total
103 //! components, the offset for a given tag, and computing the offset map, need
104 //! no change -- even if the order of the number of components change.
105 template< typename... Tags >
106 class ncomponents : public tk::tuple::tagged_tuple< Tags... > {
107 
108  private:
109  //! Create mpl::list from variadic pack, storing no. of comps with types
110  using tagsncomps = typename tk::make_list< Tags... >::type;
111 
112  public:
113  //! Remove std::vector< ncomp_type > types, i.e., keep only the tags
114  using tags = typename
115  boost::mpl::remove< tagsncomps, std::vector< ncomp_type > >::type;
116 
117  private:
118  //! Function object for zeroing all number of components
119  struct zero {
120  //! Need to store reference to host class whose data we operate on
121  ncomponents* const m_host;
122  //! Constructor: store host object pointer
123  zero( ncomponents* const host ) : m_host( host ) {}
124  //! Function call operator templated on the type that does the zeroing
125  template< typename U > void operator()( U ) {
126  //! Loop through and zero all elements of the vector for this system
127  for (auto& c : m_host->template get< U >()) c = 0;
128  }
129  };
130 
131  //! \brief Function object for computing the total number of components
132  //! (i.e., the sum of all of the number of components)
133  struct addncomp {
134  //! Need to store reference to host class whose data we operate on
135  const ncomponents* const m_host;
136  //! Internal reference used to return the total number of components
137  ncomp_type& m_nprop;
138  //! \brief Constructor: store host object pointer and initially zeroed
139  //! counter reference
140  addncomp( const ncomponents* const host, ncomp_type& nprop ) :
141  m_host( host ), m_nprop( nprop = 0 ) {}
142  //! Function call operator templated on the type that does the counting
143  template< typename U > void operator()( U ) {
144  //! Loop through and add up all elements of the vector for this system
145  for (const auto& c : m_host->template get< U >()) m_nprop += c;
146  }
147  };
148 
149  //! \brief Function object for computing the offset for a given tag (i.e.,
150  //! the sum of the number of components up to a given tag)
151  //! \details This is used to index into the data array (the equation systems
152  //! operate on during the numerical solution) and get to the beginning of
153  //! data for a given differential equation system.
154  template< typename tag >
155  struct addncomp4tag {
156  //! Need to store reference to host class whose data we operate on
157  const ncomponents* const m_host;
158  //! Internal reference used to return the offset for the tag given
159  ncomp_type& m_offset;
160  //! \brief Internal storage for the index of a system within systems
161  //! \details Example: I want the second Dirichlet system: m_c = 1.
162  //! \see offset().
163  const ncomp_type m_c;
164  //! Indicates whether the tag (eq system) was found, so it is time to quit
165  bool m_found;
166  //! \brief Constructor: store host object pointer, initially zeroed offset
167  //! reference, and system index we are looking for
168  addncomp4tag( const ncomponents* const host, ncomp_type& offset,
169  ncomp_type c ) :
170  m_host( host ), m_offset( offset = 0 ), m_c( c ), m_found( false ) {}
171  //! \brief Function call operator templated on the type that does the
172  //! offset calculation
173  template< typename U > void operator()( U ) {
174  if (std::is_same< tag, U >::value) {
175  // Make sure we are not trying to index beyond the length for this U
176  Assert( m_host->template get<U>().size() >= m_c,
177  "Indexing out of bounds in addncomp4tag!" );
178  // If we have found the tag we are looking for, we count up to the
179  // given system index (passed in via the constructor) and add those to
180  // the offset, then quit
181  for (ncomp_type c=0; c<m_c; ++c)
182  m_offset += m_host->template get<U>()[c];
183  m_found = true;
184  } else if (!m_found) {
185  // If we have not found the tag we are looking for, we add all the
186  // number of scalars for that tag to the offset
187  for (const auto& c : m_host->template get< U >()) m_offset += c;
188  }
189  }
190  };
191 
192  public:
193  //! Default constructor: set defaults to zero for all number of components
194  ncomponents() { boost::mpl::for_each< tags >( zero( this ) ); }
195 
196  //! \return Total number of components
197  ncomp_type nprop() const noexcept {
198  ncomp_type n;
199  boost::mpl::for_each< tags >( addncomp( this, n ) );
200  return n;
201  }
202 
203  //! \return offset for tag
204  //! \param[in] c Index for system given by template argument tag
205  template< typename tag >
206  ncomp_type offset( ncomp_type c ) const noexcept {
207  ncomp_type n;
208  boost::mpl::for_each< tags >( addncomp4tag< tag >( this, n, c ) );
209  return n;
210  }
211 
212  //! \return Map of offsets associated to dependent variables
213  //! \param[in] depvars Vector of vectors of dependent variables
214  OffsetMap offsetmap( const std::vector< std::vector< char > >& depvars )
215  const {
216  OffsetMap o;
217  // This is now simple, as the heavy lifting has already been done by
218  // extracting the dependent variables to a vector of vectors that is
219  // passed in. Note that for now, we simply assign linearly increasing
220  // offsets. This may be changed in the future if/when we decide to
221  // implement a user-configurable order of the data layout for the various
222  // equations.
223  ncomp_type c = 0;
224  for (const auto& vec : depvars) for (const auto& v : vec) o[ v ] = c++;
225  return o;
226  }
227 };
