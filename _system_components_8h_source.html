2 /*!
3  \file src/Control/SystemComponents.h
4  \copyright 2012-2015, J. Bakosi, 2016-2018, Los Alamos National Security, LLC.
5  \brief Operations on numbers of scalar components of systems of equations
6  \details Operations on numbers of scalar components of systems of equations,
7  e.g. multiple equation sets of a physical model or a set of (stochastic
8  ordinary or partial differential) equations of different kinds.
9 
10  _Problem:_ We are given a type that is a tk::tuple::tagged_tuple that
11  contains an arbitrary number of std::vectors of integers. The number of
12  vectors are fixed at compile-time (accessed via tags) but their (differing)
13  length is only known at run-time (after parsing user input). What we need
14  are functions that operate on this data structure and return, e.g., the
15  total number of components in the whole system or the offset in the whole
16  data structure for a given tag. The functions should thus be able to operate
17  on a list of types, i.e., a double for loop over all tags and associated
18  vectors - one at compile-time and the other one at run-time.
19 
20  _Example:_ An example is to define storage for systems of stochastic
21  differential equations as in walker::ctr::ncomps. In
22  [walker](walker.html) various types of stochastic differential
23  equations are implemented and numerically integrated in time in order to
24  compute and learn about their statistical behavior. Since different types of
25  systems can be parameterized differently, there can be several systems of
26  the same type (the number is user-defined so only known at run-time). For
27  example, it is possible to numerically integrate two systems of Dirichlet
28  equations, see DiffEq/Dirichlet.h, one with 4, the other one with 5 scalar
29  variables, parameterized differently, and estimate their arbitrary coupled
30  statistics and joint PDFs. This requires that each type of system
31  has a vector of integers storing the number of scalar variables.
32 
33  _Solution:_ Looping through elements of a tuple is done via [Boost]
34  (http://www.boost.org)'s [MetaProgramming Library (MPL)]
35  (http://www.boost.org/doc/libs/release/libs/mpl)'s [for_each]
36  (http://www.boost.org/doc/libs/release/libs/mpl/doc/refmanual/for-each.html).
37  Such operations on types happen at compile-time, i.e., the code runs inside
38  the compiler and only its result gets compiled into code to be run at
39  run-time. Advantages are abstraction and generic code that is independent
40  of the size and order of the tags in the tuple (and the associated vectors).
41  Though it is not of primary concern for this data structure, this solution
42  also generates efficient code, since part of the algorithm (for computing
43  the offset, the total number of components, etc.) runs inside the compiler.
44  All of this is kept generic, so it does not need to be changed when a new
45  pair of tag and system of equations are added to the underlying tuple.
46  Thus, _the main advantage is that changing the order of the vectors in the
47  tuple or adding new ones at arbitrary locations will not require change to
48  client-code._
49 
50  _An alternative approach:_ Of course this could have been simply done with a
51  purely run-time vector of vector of integers. However, that would not have
52  allowed a tag-based access to the different systems of equations. (Sure, one
53  can define an enum for equation indexing, but that just adds code to what is
54  already need to be changed if a change happens to the underlying data
55  structure.) As a consequence, a component in a vector of vector would have
56  had to be accessed as something like, ncomps[2][3], which is more
57  error-prone to read than ncomps< tag::dirichlet >( 3 ). Furthermore, that
58  design would have resulted in double-loops at run-time, instead of letting
59  the compiler loop over the types at compile-time and do the run-time loops
60  only for what is only known at run-time. Additionally, and most importantly,
61  _any time a new system is introduced (or the order is changed), all
62  client-code would have to be changed._
63 
64  For example client-code, see walker::Dirichlet::Dirichlet() in
65  DiffEq/Dirichlet.h, or walker::Integrator::Integrator in Walker/Integrator.h.
66 */

88 using ncomp_type = kw::ncomp::info::expect::type;

90 //! \brief Map associating offsets to dependent variables for systems
91 //! \details This map associates offsets of systems of differential !
92 //! equations in a larger data array storing dependent variables for all
93 //! scalar components of a system of systems. These offsets are where a
94 //! particular system starts and their field (or component) ids then can be
95 //! used to access an individual scalar component based on theses offsets.
96 //! \note We use a case-insensitive character comparison functor for the
97 //! offset map, since the keys (dependent variables) in the offset map are
98 //! only used to indicate the equation's dependent variable, however, queries
99 //! (using std::map's member function, find) can be fired up for both ordinary
100 //! and central moments of dependent variables (which are denoted by upper and
101 //! lower case, characters, respectively) for which the offset (for the same
102 //! dependent variable) should be the same.
103 using OffsetMap = std::map< char, ncomp_type, CaseInsensitiveCharLess >;
104 
105 //! \brief Map associating number of scalar components to dependent variables
106 //! for systems
107 //! \details This map associates the number of properties (scalar components)
108 //! of systems of differential equations for all scalar components of a
109 //! system of systems.
110 //! \note We use a case-insensitive character comparison functor to be
111 //! consistent with OffsetMap.
112 using NcompMap = std::map< char, ncomp_type, CaseInsensitiveCharLess >;

114 //! \brief Number of components storage
115 //! \details All this trickery with boost::mpl allows the code below to be
116 //! generic. As a result, adding a new component requires adding a single line
117 //! (a tag and its type) to the already existing list, see typedefs 'ncomps'.
118 //! The member functions, doing initialization, computing the number of total
119 //! components, the offset for a given tag, and computing the offset map, need
120 //! no change -- even if the order of the number of components change.
121 template< typename... Tags >
122 class ncomponents : public tk::tuple::tagged_tuple< Tags... > {
123 
124  private:
125  //! Create mpl::list from variadic pack, storing no. of comps with types
126  using tagsncomps = typename tk::make_list< Tags... >::type;
127 
128  public:
129  //! Remove std::vector< ncomp_type > types, i.e., keep only the tags
130  using tags = typename
131  boost::mpl::remove< tagsncomps, std::vector< ncomp_type > >::type;
132 
133  private:
134  //! Function object for zeroing all number of components
135  struct zero {
136  //! Need to store reference to host class whose data we operate on
137  ncomponents* const m_host;
138  //! Constructor: store host object pointer
139  zero( ncomponents* const host ) : m_host( host ) {}
140  //! Function call operator templated on the type that does the zeroing
141  template< typename U > void operator()( U ) {
142  //! Loop through and zero all elements of the vector for this system
143  for (auto& c : m_host->template get< U >()) c = 0;
144  }
145  };
146 
147  //! \brief Function object for computing the total number of components
148  //! (i.e., the sum of all of the number of components)
149  struct addncomp {
150  //! Need to store reference to host class whose data we operate on
151  const ncomponents* const m_host;
152  //! Internal reference used to return the total number of components
153  ncomp_type& m_nprop;
154  //! \brief Constructor: store host object pointer and initially zeroed
155  //! counter reference
156  addncomp( const ncomponents* const host, ncomp_type& nprop ) :
157  m_host( host ), m_nprop( nprop = 0 ) {}
158  //! Function call operator templated on the type that does the counting
159  template< typename U > void operator()( U ) {
160  //! Loop through and add up all elements of the vector for this system
161  for (const auto& c : m_host->template get< U >()) m_nprop += c;
162  }
163  };
164 
165  //! \brief Function object for computing the offset for a given tag (i.e.,
166  //! the sum of the number of components up to a given tag)
167  //! \details This is used to index into the data array (the equation systems
168  //! operate on during the numerical solution) and get to the beginning of
169  //! data for a given differential equation system.
170  template< typename tag >
171  struct addncomp4tag {
172  //! Need to store reference to host class whose data we operate on
173  const ncomponents* const m_host;
174  //! Internal reference used to return the offset for the tag given
175  ncomp_type& m_offset;
176  //! \brief Internal storage for the index of a system within systems
177  //! \details Example: I want the second Dirichlet system: m_c = 1.
178  //! \see offset().
179  const ncomp_type m_c;
180  //! Indicates whether the tag (eq system) was found, so it is time to quit
181  bool m_found;
182  //! \brief Constructor: store host object pointer, initially zeroed offset
183  //! reference, and system index we are looking for
184  addncomp4tag( const ncomponents* const host, ncomp_type& offset,
185  ncomp_type c ) :
186  m_host( host ), m_offset( offset = 0 ), m_c( c ), m_found( false ) {}
187  //! \brief Function call operator templated on the type that does the
188  //! offset calculation
189  template< typename U > void operator()( U ) {
190  if (std::is_same< tag, U >::value) {
191  // Make sure we are not trying to index beyond the length for this U
192  Assert( m_host->template get<U>().size() >= m_c,
193  "Indexing out of bounds in addncomp4tag!" );
194  // If we have found the tag we are looking for, we count up to the
195  // given system index (passed in via the constructor) and add those to
196  // the offset, then quit
197  for (ncomp_type c=0; c<m_c; ++c)
198  m_offset += m_host->template get<U>()[c];
199  m_found = true;
200  } else if (!m_found) {
201  // If we have not found the tag we are looking for, we add all the
202  // number of scalars for that tag to the offset
203  for (const auto& c : m_host->template get< U >()) m_offset += c;
204  }
205  }
206  };

208  //! Function object for creating offset map: std::map< depvar, offset >
209  template< class InputDeck >
210  struct genOffsetMap {
211  //! Reference to input deck to operate on
212  const InputDeck& deck;
213  //! Internal reference to map we build
214  OffsetMap& map;
215  //! Constructor: store reference to input deck and offset map
216  genOffsetMap( const InputDeck& d, OffsetMap& m ) : deck( d ), map( m ) {}
217  //! \brief Functional call operator templated on the type that computes
218  //! the offset map for type U.
219  //! \details There can be multiple systems of the same equation type,
220  //! differentiated by a different dependent variable.
221  template< typename U > void operator()( U ) const {
222  const auto& depvar = deck.template get< tag::param, U, tag::depvar >();
223  ncomp_type c = 0;
224  const auto& ncomps = deck.template get< tag::component >();
225  for (auto v : depvar) map[ v ] = ncomps.template offset< U >( c++ );
226  }
227  };
228 
229  //! \brief Function object for creating number of properties (scalar
230  //! components) map: std::map< depvar, offset >
231  template< class InputDeck >
232  struct genNcompMap {
233  //! Reference to input deck to operate on
234  const InputDeck& deck;
235  //! Internal reference to map we build
236  NcompMap& map;
237  //! Constructor: store reference to input deck and offset map
238  genNcompMap( const InputDeck& d, NcompMap& m ) : deck( d ), map( m ) {}
239  //! \brief Functional call operator templated on the type that computes
240  //! the number of properties (scalar components) map for type U.
241  //! \details There can be multiple systems of the same equation type,
242  //! differentiated by a different dependent variable.
243  template< typename U > void operator()( U ) const {
244  const auto& depvar = deck.template get< tag::param, U, tag::depvar >();
245  const auto& ncomps = deck.template get< tag::component >();
246  ncomp_type c = 0;
247  for (auto v : depvar) map[ v ] = ncomps.template get< U >().at( c++ );
248  }
249  };
250 
251  public:
252  //! Default constructor: set defaults to zero for all number of components
253  ncomponents() { boost::mpl::for_each< tags >( zero( this ) ); }
254 
255  //! \return Total number of components
256  ncomp_type nprop() const noexcept {
257  ncomp_type n;
258  boost::mpl::for_each< tags >( addncomp( this, n ) );
259  return n;
260  }
261 
262  //! \return offset for tag
263  //! \param[in] c Index for system given by template argument tag
264  template< typename tag >
265  ncomp_type offset( ncomp_type c ) const noexcept {
266  ncomp_type n;
267  boost::mpl::for_each< tags >( addncomp4tag< tag >( this, n, c ) );
268  return n;
269  }
270 
271  //! Compute map of offsets associated to dependent variables
272  //! \param[in] deck Input deck to operate on
273  //! \return Map of offsets associated to dependent variables
274  template< class InputDeck >
275  OffsetMap offsetmap( const InputDeck& deck ) const {
276  OffsetMap map;
277  boost::mpl::for_each< tags >( genOffsetMap< InputDeck >( deck, map ) );
278  return map;
279  }
280 
281  //! \brief Compute map of number of properties (scalar components)
282  //! associated to dependent variables
283  //! \param[in] deck Input deck to operate on
284  //! \return Map of number of properties associated to dependent variables
285  template< class InputDeck >
286  NcompMap ncompmap( const InputDeck& deck ) const {
287  NcompMap map;
288  boost::mpl::for_each< tags >( genNcompMap< InputDeck >( deck, map ) );
289  return map;
290  }
291 };
