57 //! Global-scope data. Initialized by the main chare and distibuted to all PEs by
58 //! the Charm++ runtime system. Though semantically not const, all these global
59 //! data should be considered read-only. See also http://charm.cs.illinois.edu/
60 //! manuals/html/charm++/manual.html. The data below is global-scope because they
61 //! must be available to all PEs which could be on different machines. In a
62 //! previous non-Charm++ design, most of this data was held at class-level, but
63 //! since the generators in g_rng must be possible to be called from
64 //! global-scope, as external generators to TestU01, it is easier to make g_rng
65 //! global-scope, as well the additional data required to initialize it,
66 //! contained in g_inputdeck (storing all parsed user input). This is required
67 //! for serializing during migration of g_rng across the network.
68 //!
69 //! Note that the container (std::map) holding tk::RNG objects uses value
70 //! semantics which is safer and generally less error-prone than reference
71 //! semantics. At the same time tk::RNG is used in a polymorphic fashion with
72 //! various classes that adhere to the concepts required by Concept defined
73 //! inside tk::RNG. tk::RNG does not define a default, i.e., non-templated
74 //! constructor, since then the "derived" class object could not be initialized
75 //! rendering the class tk::RNG empty-constructed, which invites abuse and
76 //! ill-defined behavior. As such, the "derived" class type comes through the
77 //! constructors and thus would not be available for a pack/unpack migrator
78 //! required by Charm++ from within. Templating the class tk::RNG is not an
79 //! option since then we could not hold tk::RNG objects in a simple std::vector.
80 //! As a result of the above requirements, the tk::RNG objects in g_rng are
81 //! migrated (here in global-scope) by reinstantiating RNGStack, which
82 //! reinstatiates the RNG factory, from which the RNGs selected by the user are
83 //! instantiated.
84 //!
85 //! Note also that RNGFactory associates tk::ctr::RNG ids (enum class values) to
86 //! function pointers (std::function objects pointing to tk::RNG constructors
87 //! bound with their arguments). Since function pointers cannot simply be
88 //! serialized and migrated via the network, they must also be recreated on
89 //! remote machines. This initial migration of global-scope data is done by the
90 //! Charm++ runtime once the main chare constructor is finished -- see the
91 //! RNGTestDriver constructor, which initializes the data required for the
92 //! migration).

128 //! Pack/Unpack selected RNGs. This Pack/Unpack method (re-)creates the full RNG
129 //! stack since it needs to (re-)bind function pointers on different processing
130 //! elements. Therefore we circumvent Charm's usual pack/unpack for this type,
131 //! and thus sizing does not make sense: sizing is a no-op. We could initialize
132 //! the stack in RNGTestDriver's constructor and let this function re-create the
133 //! stack only when unpacking, but that leads to repeating the same code twice:
134 //! once in RNGTestDriver's constructor, once here. Another option is to use
135 //! this pack/unpack routine to both initially create (when packing) and to
136 //! re-create (when unpacking) the stack, which eliminates the need for
137 //! pre-creating the object in RNGTestDriver's constructor and therefore
138 //! eliminates the repeated code. This explains the guard for sizing: the code
139 //! below is called for packing only (in serial) and packing and unpacking (in
140 //! parallel).
141 inline
142 void operator|( PUP::er& p, std::map< tk::ctr::RawRNGType, tk::RNG >& rng ) {
143  if (!p.isSizing()) {
144  tk::RNGStack stack(
145  #ifdef HAS_MKL
146  g_inputdeck.get< tag::param, tag::rngmkl >(),
147  #endif
148  g_inputdeck.get< tag::param, tag::rngsse >() );
149  rng = stack.selected( g_inputdeck.get< tag::selected, tag::rng >() );
150  }
151 }

171 //! \brief Charm++ main chare for the random number generator test suite
172 //! executable, rngtest.
173 //! \details Note that this object should not be in a namespace.
174 class Main : public CBase_Main {
175 
176  public:
177  //! \brief Constructor
178  //! \details The main chare constructor is the main entry point of the
179  //! program, called by the Charm++ runtime system. The constructor does
180  //! basic initialization steps, e.g., parser the command-line, prints out
181  //! some useful information to screen (in verbose mode), and instantiates
182  //! a driver. Since Charm++ is fully asynchronous, the constructor
183  //! usually spawns asynchronous objects and immediately exits. Thus in the
184  //! body of the main chare constructor we fire up an 'execute' chare,
185  //! which then calls back to Main::execute(). Finishing the main chare
186  //! constructor the Charm++ runtime system then starts the
187  //! network-migration of all global-scope data (if any). The execute chare
188  //! calling back to Main::execute() signals the end of the migration of
189  //! the global-scope data. Then we are ready to execute the driver. Since
190  //! the random number generator test suite is parallel and asynchronous,
191  //! its driver fires up additional Charm++ chare objects which then call
192  //! back to Main::finalize() at some point in the future when all work has
193  //! been finished. finalize() then exits by calling Charm++'s CkExit(),
194  //! shutting down the runtime system.
195  //! \see http://charm.cs.illinois.edu/manuals/html/charm++/manual.html
196  Main( CkArgMsg* msg )
197  try :
198  m_cmdline(),
199  // Parse command line into m_cmdline using default simple pretty printer
200  m_cmdParser( msg->argc, msg->argv, tk::Print(), m_cmdline ),
201  // Create pretty printer initializing output streams based on command line
202  m_print( m_cmdline.get< tag::verbose >() ? std::cout : std::clog ),
203  // Create RNGTest driver
204  m_driver( tk::Main< rngtest::RNGTestDriver >
205  ( msg->argc, msg->argv,
206  m_cmdline,
207  tk::HeaderType::RNGTEST,
208  RNGTEST_EXECUTABLE,
209  m_print ) ),
210  m_timer(1), // Start new timer measuring the total runtime
211  m_timestamp()
212  {
213  delete msg;
214  mainProxy = thisProxy;
215  // Fire up an asynchronous execute object, which when created at some
216  // future point in time will call back to this->execute(). This is
217  // necessary so that this->execute() can access already migrated
218  // global-scope data.
219  CProxy_execute::ckNew();
220  // Start new timer measuring the migration of global-scope data
221  m_timer.emplace_back();
222  } catch (...) { tk::processExceptionCharm(); }
223 
224  void execute() {
225  try {
226  m_timestamp.emplace_back("Migrate global-scope data", m_timer[1].hms());
227  m_driver.execute(); // fires up async chares
228  } catch (...) { tk::processExceptionCharm(); }
229  }
230 
231  void finalize() {
232  try {
233  if (!m_timer.empty()) {
234  m_timestamp.emplace_back( "Total runtime", m_timer[0].hms() );
235  m_print.time( "Timers (h:m:s)", m_timestamp );
236  m_print.endpart();
237  }
238  } catch (...) { tk::processExceptionCharm(); }
239  // Tell the Charm++ runtime system to exit
240  CkExit();
241  }
