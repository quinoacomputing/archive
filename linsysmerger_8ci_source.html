72  // SDAG code follows. See http://charm.cs.illinois.edu/manuals/html/
73  // charm++/manual.html, Sec. "Structured Control Flow: Structured Dagger".
74 
75  // High-level overview of the dependency and asynchronous call structure
76  // ---------------------------------------------------------------------
77  //
78  // Directed Acyclic Graph (DAG):
79  // -----------------------------
80  //
81  // See src/LinSys/LinSysMerger.h.
82  //
83  // Interpretation of the DAG in src/LinSys/LinSysMerger.h
84  // ------------------------------------------------------
85  // Control flow is from left to right and top to bottom.
86  //
87  // We start out with all worker chares contributing their porition of the
88  // global row IDs that each work on. This is labelled ChRow. The worker
89  // chares also contribute, in parallel, their portion of the global node
90  // ID lists at which they can set boundary conditions, labelled, ChBCs.
91  // Once both row IDs and BC node lists are received on LinSysMerger, we do
92  // a global reduction to Transport::rowcomplete() which spawns two
93  // broadcasts: (1) LinSysMerger::rowsreceived(), and (2) worker::init().
94  //
95  // LinSysMerger::rowsreceived() path: The member function
96  // tk::LinSysMerger::rowsreceived(), labeled Ver in the task-graph,
97  // triggers and performs the following tasks: (1) Verification that
98  // ensures consistent global row IDs. This step only happens in DEBUG
99  // mode: VerRow is simply an Assert in member function rowsreceived). (2)
100  // The second step in tk::LinSysMerger::rowsreceived() initiates is
101  // building Hypre data from the global row indices received, labeled
102  // HypreRow.
103  //
104  // Worker::init() path: Once the row IDs are contributed,
105  // tk::Transporter::rowcomplete(), not only spaws
106  // tk::LinSysMerger::rowsreceived() but also issues a broadcast to the
107  // worker chares to start initializing their system. This is started with
108  // a call to the init() member function of the worker. The tasks done in
109  // init() include setting initial and boundary conditions, sending
110  // unknown/solution vectors for assembly to LinSysMerger, start computing
111  // the minimum time step size, outputing initial field data to file, start
112  // computing the left hand sides for both the primary and auxiliary linear
113  // systems, and sending all this for linear system assembly. The
114  // contribution goes to those Linear System Merger group branch (one per
115  // CPU) the workers happen to reside on. These steps are denoted by ChSol,
116  // ChLhs, and ChAuxLhs in the graph above, and the contributions/assembly
117  // is performed by the member functions charesol(), charelhs(), and
118  // chareauxlhs(), respectively.
119  //
120  // Among other tasks, worker::init(), discussed above, also start
121  // computing the minimum time step sizes based on all PDEs integrated.
122  // Once a worker has finished with that task, it sends it contribution of
123  // the smallest dt for computing a global minimum in Transporter::dt(),
124  // which then spawns a broadcast to the workers to start advancing their
125  // PDEs, which consists of computing the right hand sides for both the
126  // primary system and the auxiliary system, ChRhs, ChAuxRhs. These
127  // right hand sides can only be comnputed if the dt is available.
128  //
129  // Once the collection of the right hand side vector, ChRhs, is done, we
130  // set boundary conditions on the RHS vector, RhsBC. Once the collection
131  // of the left hand side matrix, ChLhs, is done, we set boundary
132  // conditions on the LHS matrix, LhsBC.
133  //
134  // Primary solution only: Once each of the tasks, ChSol, LhsBC, and RhsBC,
135  // (not all at the same time but separately) are done, we continue by
136  // converting these data structures to a format that Hypre expects, done
137  // in hypresol(), hyprelhs(), and hyprerhs(), denoted by similars labels
138  // in the graph. These functions basically flatten the C++ data
139  // structures to C-style arrays and linked arrays (for the matrix). Note
140  // that the left and right hand sides can only begin to be converted to
141  // Hypre data format if the boundary conditions have also been applied on
142  // each.
143  //
144  // Primary solution only: Once the tasks, HypreSol, HypreLhs, HypreRhs,
145  // (not all at the same time but separately) are done and the Hypre data
146  // structure for the row IDs are complete, we continue by assigning the
147  // flattened data structures (more precisely, their pointers) to the Hypre
148  // vectors and Hypre matrix. These are denoted FillSol, FillLhs, and
149  // FillRhs in the DAG and correspond to the member functions sol(), lhs(),
150  // and rhs().
151  //
152  // Primary solution only: Once the tasks, FillSol, FillLhs, FIllRhs, (not
153  // all at the same time, but separately) are, we continue by calling the
154  // Hypre "assembly" routines for the solution vector, the left hand side
155  // matrix, and the right hand side vector. These are denoted by AsmSol,
156  // AsmLhs, and AsmRhs in the DAG, and performed by the member functions
157  // assemblesol(), assemblelhs(), and assemblerhs(), respectively.
158  //
159  // Primary solution only: Once the assembly of the solution (unknown)
160  // vector, the left hand side matrix, and the right hand side vector are
161  // all done (on a CPU), labels AsmSol, AsmLhs, AsmRhs, respectively, we
162  // call the solve() member function which solves the primary linear system
163  // by calling Hypre and also updates the primary solution, i.e.,
164  // propagates the new solution back to the worker chares, labeled Upd.
165  //
166  // Auxiliary solution only: The auxiliary solution is performed once the
167  // boundary conditions have been set on the primary right hand side vector
168  // (RhsBC), the auxiliary rhs vector, combined with primary the rhs
169  // (AuxRhs) has been ready, and the auxiliary left hand side (AuxLhs) are
170  // all complete. The solution of the auxiliary system is a simple
171  // back-substitution, since the system is assumed diagonal. Note also,
172  // that due to the way FluxCorrector::aec() computes the antidiffusive
173  // element contributions and the way it applies the limiter in
174  // FluxCorrector::lim(), there is no need to set Dirichlet boundary
175  // conditions on the auxiliary system. See more details in
176  // FluxCorrector::aec().
177  //
178  // A word on multiple invokation of SDAG triggers: As the commit message
179  // of b79ea29 explains, the "triggers", such as hyprerow_complete(),
180  // enumerated below, at threir call site should be thought of as messages
181  // being put in some message queue and consumed at when clauses wherever
182  // they appear. If they appear in multiple when clauses, they consume
183  // multple messages, thus they have to be triggered multiple times. See
184  // also
185  // https://lists.cs.illinois.edu/lists/arc/charm/2016-09/msg00002.html.
186  //
187  // As discussed in src/LinSys/LinSysMerger.h, the SDAG logic discussed
188  // here is the same regardless whether an auxiliary linear solution is
189  // performed.
