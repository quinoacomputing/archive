31  // High-level overview of the dependency and asynchronous call structure
32  // ---------------------------------------------------------------------
33  //
34  // Directed Acyclic Graph (DAG): DAG legend:
35  // ----------------------------- AdvP - advance particles
36  // OrdM - estimate ordinary moments
37  // AdvP -- OrdM -- CenM -- OutS CenM - estimate central moments
38  // | \ \ | OrdP - estimate ordinary PDFs
39  // \ OrdP CenP | CenP - estimate central PDFs
40  // \ \ \ | OutS - output statistics (mom)
41  // \ ------ OutP -- EvT OutP - output PDFs
42  // \ / EvT - Evaluate time step
43  // ---- NoSt ------ / NoSt - No stats & nor PDFs
44  //
45  // Interpretation of the above DAG
46  // -------------------------------
47  // Control flow is from left to right and top to bottom. In the discussion
48  // below, concentrate first on the first three lines and ignore NoSt for
49  // the moment - those will be discussed at the very end.
50  //
51  // After initialization a time step starts by advancing all particles
52  // (AdvP, Integrator::advance()). When that is finished (no global
53  // synchronization), each PE starts with the estimation of its portion of
54  // the ordinary moments (OrdM). When a PE finished with its part of the
55  // estimation of the ordinary moments, it calls back to its host,
56  // Distributor, which collects statistics from all PEs. When a PE has
57  // finished its accumulation of ordinary moments and has sent its
58  // contribution to its host, it immediately continues with estimation of
59  // ordinary PDFs (OrdP) (if any), and does not wait for its host.
60  //
61  // Understanding 'wait4ord'. Once the host has finished collecting
62  // partially accumulated ordinary moment (sums) from all PEs, it signals
63  // Charm++ SDAG runtime that all ordinary moments are updated and the
64  // estimation of central moments and PDFs can start. This is a necessary
65  // synchronization point (central moments can only be estimated about some
66  // ordinary moments, i.e., means), realized by two steps: (1) starting off
67  // the entry 'wait4ord' (before the time step), which waits for the SDAG
68  // signal 'estimateOrdDone', sent by the host. Once the SDAG signal is in,
69  // as step (2), the request to start the accumulation of the central
70  // moments (CenM) are asynchronously fired off to all PEs followed by a
71  // similar request for the central PDFs (CenP) (if any). Once a PE has
72  // finished its part of the accumuluation of the central moments, it sends
73  // its portion of the sums back to its host which collects from all PEs
74  // and finishes the estimation.
75  //
76  // Understanding 'wait4cen'. Once the host has finished collecting
77  // partially accumulated central moment (sums) from all PEs, it signals
78  // Charm++ SDAG runtime that all central moments are updated and the
79  // statistics (moments) are ready to be written to disk (OutS) (if
80  // scheduled in this step). This is also a necessary synchronization point
81  // and done similarly to the one discussed above, but this time by
82  // 'wait4cen' and 'estimateCenDone'. Since there may not be any PDFs
83  // estimated, having finished estimating the central moments may finish
84  // the time step and 'evalulateTime' can be called (EvT). If PDFs are
85  // estimated evaluateTime is called only after the PDFs have been
86  // estimated.
87  //
88  // Understanding 'wait4pdf'. Once a PE has finished its part of the
89  // accumuluation of the central PDFs (started from 'wait4ord'), it sends
90  // its portion of the sums back to its host which collects from all PEs
91  // and finishes the estimation. The host then signals Charm++ SDAG
92  // runtime that all central PDFs have been estimated. This is also a
93  // necessary synchronization point and done similarly to the ones
94  // discussed above, but this time by 'wait4pdf' waiting for two signals,
95  // 'estimateOrdPDFDone' and 'estimateCenPDFDone'. Both the estimation of
96  // ordinary and central PDFs must have completed in order to be possible
97  // to be written out to disk (OutP) (if scheduled in this time step).
98  // According to the logical 'and' relationship here, two 'when' keywords
99  // are given in 'wait4pdf'. This enables overlapping OrdP with everything
100  // else discussed thus far.
101  //
102  // The resulting algorithm for a single time step enables overlapping
103  // computations, communications as well as I/O. The algorithm has two
104  // global synchronization points: (1) it must wait for all particles to be
105  // advanced (AdvP), and (2) the final evaluateTime() (EvT) can only happen
106  // once all required statistics and PDFs have been estimated and
107  // optionally written to disk. Furthermore, there are three partial
108  // synchronization points: (1) since central moments and central PDFs can
109  // only be estimated about already estimated (and collected) ordinary
110  // moments, there is a barrier at OrdM, but this is overlapped with OrdP,
111  // (2) since statistics (moments) can only be output to disk if they all
112  // have been estimated and collected, there is a barrier at CenM, but this
113  // is overlapped with OrdP, CenP, and OutP, and (3) since PDFs can only be
114  // output to disk if they all have been estimated and collected, there is
115  // a barrier at CenP, but this is overlapped with OrdP, CenM, and OutS.
116  //
117  // NoSt in the graph signals a potential shortcut which is activated if
118  // there are no statistics nor PDFs need to be estimated. In that case,
119  // after advancing the particles, control flow just to evaluating the time
120  // step, skipping over several synchronization points.
121  //
122  // Similar to NoSt, the estimateion of the PDFs can also be potentially
123  // skipped. This happens when either the the user did not request any PDF
124  // estimation or the interval for estimating and outputing PDFs is such
125  // that at the given time step PDFs are not estimated. (Estimation of
126  // PDFs, especially multi-dimensional ones, can be quite costly, and thus
127  // PDF estimation is recommended only at certain intervals, i.e., not
128  // every time step.) If the shortcut is activated due to no PDF esimation,
129  // the control flow in wait4cen() skips ahead to evaluate time, i.e.,
130  // finish the time step, after the central moments (if any) have been
131  // estimated. Similarly, of the shortcut is activated due to no PDF
132  // estimation, the control flow in wait4ord() skips the estimation of
133  // central PDFs.
134 
135  // Note that estimating the ordinary moments and the ordinary PDFs are
136  // started in Integrator::accumulateOrd(), and when their accumulation
137  // step is done, a Charm++ group, Collector, collects those contributions
138  // from Integrators that happened to be on its PE. This collection is done
139  // via Collector::chareOrd(), which aggregates the partial sums (for the
140  // given PE) and forwards them to the host, Distributor. While the
141  // accumulation of the ordinary moments and the ordinary PDFs happen
142  // serially in Integrator::accumulateOrd(), their aggregation from
143  // Collectors to Distributor happen asynchronously. The same procedure is
144  // followed for the central moments and the central PDFs. If PDFs are not
145  // collected in a time step, the PDF containers are simply empty. This
146  // simplifies asynchronous logic.
147 
148  // SDAG wait-for: wait for ordinary moments to have been estimated
149  entry void wait4ord() {
150  when estimateOrdDone() serial "accumulateCen" {
151  // Start accumulating sums for central moments
152  m_intproxy.accumulateCen( m_it, m_ordinary );
153  }
154  };

156  // SDAG wait-for: wait for PDFs to have been estimated, the two 'when'
157  // statements are undestood with a logical AND between them
158  entry void wait4pdf() {
159  when estimateCenDone(),
160  estimateOrdPDFDone(),
161  estimateCenPDFDone() serial "outPDF"
162  {
163  outStat(); // Output statistics to file
164  outPDF(); // output PDFs to file
165  evaluateTime(); // evaluate time step, compute new time step
166  }
167  };
