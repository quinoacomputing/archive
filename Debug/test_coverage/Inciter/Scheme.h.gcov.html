       5                 :            :   \brief     Generic forwarding interface to discretization proxies
       6                 :            :   \details   This file defines a generic interface to discretization proxies.
       7                 :            : 
       8                 :            :     The purpose of this class is to hide, behind a single type, different
       9                 :            :     Charm++  proxy types that model a single concept, i.e., define some common
      10                 :            :     functions as Charm++ entry methods that can be used in either a broadcast
      11                 :            :     and/or in a way of addressing a single array element. As a result, member
      12                 :            :     functions can be invoked by client code without knowing the underlying type
      13                 :            :     or any specifics to the underlying differences of the classes that model the
      14                 :            :     same concept, i.e., expose the same member functions. The idea is very
      15                 :            :     similar to inheritance and runtime polymorphism with base classes and
      16                 :            :     virtual functions: some member functions and data are common to all types
      17                 :            :     modeled (and thus are not repeated and/or copied), while some are specific.
      18                 :            :     A difference is that the "base" and "child" classes are Charm++ proxies.
      19                 :            :     Note that while Charm++ does support inheritance and runtime polymorphism
      20                 :            :     with chare arrays, we still prefer the implementation below because it uses
      21                 :            :     entirely value semantics (inside and in client code) and thus it keeps the
      22                 :            :     complexity of the dispatch behind this class and does not expose it client
      23                 :            :     code.
      24                 :            : 
      25                 :            :     The advantages of this class over traditional runtime polymorphism are (1)
      26                 :            :     value semantics (both internally and to client code), (2) not templated,
      27                 :            :     and (3) PUPable, i.e., an instance of Scheme can be sent across the network
      28                 :            :     using Charm++'s pup framework. Also, since the class only holds a couple of
      29                 :            :     chare proxies, it is extremely lightweight.
      30                 :            : 
      31                 :            :     Example usage from client code:
      32                 :            : 
      33                 :            :     \code{.cpp}
      34                 :            :       // Instantiate a Scheme object
      35                 :            :       Scheme s( ctr::SchemeType::CG );  // see Control/Inciter/Options/Scheme.h
      36                 :            : 
      37                 :            :       // Call a member function entry method in broadcast fashion
      38                 :            :       s.coord< tag::bcast >( ... );     // equivalent to proxy.coord( ... );
      39                 :            : 
      40                 :            :       // Call a member function entry method in addressing a single array
      41                 :            :       // element
      42                 :            :       s.coord< tag::elem >( 0, ... );   // equivalent to proxy[0].coord( ... );
      43                 :            : 
      44                 :            :       // Broadcast to a member function with optinoal CkEntryOptions object
      45                 :            :       CkEntryOptions opt;
      46                 :            :       s.coord< tag::bcast >( ..., opt );     // proxy.coord( ..., opt );
      47                 :            : 
      48                 :            :       // Address array element with optinoal CkEntryOptions object
      49                 :            :       s.coord< tag::elem >( 0, ..., opt );   // proxy[0].coord( ..., opt );
      50                 :            :     \endcode
      51                 :            : 
      52                 :            :     Organization, implementation details, end extension of the class:
      53                 :            : 
      54                 :            :     Scheme, via inheriting from SchemeBase, contains two Charm++ proxies:
      55                 :            :     discproxy and proxy. The former contains data and functionality common to
      56                 :            :     all discretizations, and this can be considered as an equivalent to a base
      57                 :            :     class in the OOP sense. The latter, proxy, contains data and functionality
      58                 :            :     specific to a particular discretization. When instantiated, Scheme is
      59                 :            :     configured for a single specific discretization which must be selected from
      60                 :            :     the list of types in SchemeBase::Proxy.
      61                 :            : 
      62                 :            :     The underlying type of proxy is a variant, which allows storing exactly one
      63                 :            :     object. A variant is a type-safe union. An instance of a variant at any
      64                 :            :     given time either holds a value of one of its alternative types. Read more
      65                 :            :     on std::variant or boost::variant on how they work.
      66                 :            : 
      67                 :            :     All new member functions that comprise of the concept of the underlying
      68                 :            :     proxies, i.e., the interface, must be defined in Scheme. Whereas common
      69                 :            :     data, functionality, as well as the list of the proxy types that can be
      70                 :            :     configured are defined in SchemeBase. Adding a new forwarding function
      71                 :            :     either as a broadcast or addressing a particular chare array element can be
      72                 :            :     done by simply copying an existing (similar) one and modifying what
      73                 :            :     underlying function (entry method) it calls. The ones that forward to
      74                 :            :     discproxy are grouped first, while the ones that forward to the specific
      75                 :            :     proxy are listed as second. Using SFINAE, multiple overloads are (and can
      76                 :            :     be) defined for a single function, depending on (1) whether it is a
      77                 :            :     broadcast or addressing an array element, (2) whether it takes an optional
      78                 :            :     (default) last argument, usually used for passing a CkEntryOptions object.
      79                 :            :     You can see the Charm++-generated .decl.h file to see what (if any) default
      80                 :            :     arguments a particular entry method may take.
      81                 :            : 
      82                 :            :     Currently, forwarding functions are defined for two types entry method
      83                 :            :     calls: broadcasts, i.e., proxy.fn(), and addressing a particular element,
      84                 :            :     i.e., proxy[x].fn(). Another third might be useful to add in the future and
      85                 :            :     that is addressing an entry method behind a section proxy. As we have not
      86                 :            :     used section proxies so far, this is not yet implemented, but if necessary,
      87                 :            :     it should be relatively straightforward to do.
      88                 :            : 
      89                 :            :     Extending this class to other discretization schemes is done entirely in
      90                 :            :     SchemeBase. Adding a new discretization scheme amounts to, at the minimum:
      91                 :            :     (1) Adding a new type of Charm++ chare array proxy to SchemeBase::Proxy,
      92                 :            :     (2) Adding a new type of Charm++ chare array element proxy to
      93                 :            :         SchemeBase::ProxyElem, and
      94                 :            :     (3) Adding a new branch to the if test in SchemeBase's constructor.
      95                 :            : 
      96                 :            :     Implementation details: All forwarding calls are implemented taking a
      97                 :            :     variadic parameter pack, which can take any number of arguments (including
      98                 :            :     zero) and use perfect forwarding to pass the arguments to the entry method
      99                 :            :     calls. This way the code remains generic, easy to modify, and the compiler
     100                 :            :     automatically adjusts the generated forwarding calls if the types and/or
     101                 :            :     number of the arguments to any of the entry methods change. One exception to
     102                 :            :     this is those forwarding calls that deal with default arguments, allowing
     103                 :            :     for passing CkEntryOptions objects. There the number of arguments are
     104                 :            :     hard-coded in the SFINAE construct, but should also be straightforward to
     105                 :            :     modify if necessary.
     106                 :            : 
     107                 :            :     The functors named as call_* are used to dispatch (at compile time) entry
     108                 :            :     method calls behind proxy, whose type is different depending on what
     109                 :            :     specific discretization type is configured in the constructor. All common
     110                 :            :     functionality in the call_* functors are lifted over to SchemeBase::Call.
     111                 :            :     This helps keeping the function-call-specific code in Scheme minimal and
     112                 :            :     reuses the generic part in SchemeBase.
     113                 :            : 
     114                 :            :     Note that another way of doing the dispatch, that is now done using the
     115                 :            :     call_* functors, could have been implemented using a (compile-, or runtime)
     116                 :            :     associative container storing std::function objects which would store
     117                 :            :     pre-bound function arguments. That would work, but there are three problems
     118                 :            :     with such an approach: (1) std::function is not obvious how to pup, i.e.,
     119                 :            :     send across the network, (2) std::bind cannot currently be used to bind a
     120                 :            :     variadic number arguments and thus the bind calls would not be very generic,
     121                 :            :     and (3) a runtime associative container would take additional state.
     122                 :            : */
