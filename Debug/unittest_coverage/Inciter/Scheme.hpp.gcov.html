       9             :   \details   This file defines a generic interface to discretization proxies.
      10             : 
      11             :     The purpose of this class is to hide, behind a single type, different
      12             :     Charm++  proxy types that model a single concept, i.e., define some common
      13             :     functions as Charm++ entry methods that can be used in either a broadcast
      14             :     and/or in a way of addressing a single array element. As a result, member
      15             :     functions can be invoked by client code without knowing the underlying type
      16             :     or any specifics to the underlying differences of the classes that model the
      17             :     same concept, i.e., expose the same member functions. The idea is very
      18             :     similar to inheritance and runtime polymorphism with base classes and
      19             :     virtual functions: some member functions and data are common to all types
      20             :     modeled (and thus are not repeated and/or copied), while some are specific.
      21             :     A difference is that the "base" and "child" classes are Charm++ proxies.
      22             :     Note that while Charm++ does support inheritance and runtime polymorphism
      23             :     with chare arrays, we still prefer the implementation below because it uses
      24             :     entirely value semantics (inside and in client code) and thus it keeps the
      25             :     complexity of the dispatch behind this class and does not expose it client
      26             :     code.
      27             : 
      28             :     The advantages of this class over traditional runtime polymorphism are (1)
      29             :     value semantics (both internally and to client code), (2) not templated,
      30             :     and (3) PUPable, i.e., an instance of Scheme can be sent across the network
      31             :     using Charm++'s pup framework. Also, since the class only holds a couple of
      32             :     chare proxies, it is extremely lightweight.
      33             : 
      34             :     Example usage from client code:
      35             : 
      36             :     \code{.cpp}
      37             :       // Instantiate a Scheme object
      38             :       Scheme s( ctr::SchemeType::DG );  // see Control/Inciter/Options/Scheme.h
      39             : 
      40             :       // Call a member function entry method in broadcast fashion
      41             :       s.coord< tag::bcast >( ... );     // equivalent to proxy.coord( ... );
      42             : 
      43             :       // Call a member function entry method in addressing a single array
      44             :       // element
      45             :       s.coord< tag::elem >( 0, ... );   // equivalent to proxy[0].coord( ... );
      46             : 
      47             :       // Broadcast to a member function with optinoal CkEntryOptions object
      48             :       CkEntryOptions opt;
      49             :       s.coord< tag::bcast >( ..., opt );     // proxy.coord( ..., opt );
      50             : 
      51             :       // Address array element with optinoal CkEntryOptions object
      52             :       s.coord< tag::elem >( 0, ..., opt );   // proxy[0].coord( ..., opt );
      53             :     \endcode
      54             : 
      55             :     Organization, implementation details, end extension of the class:
      56             : 
      57             :     Scheme, via inheriting from SchemeBase, contains two Charm++ proxies:
      58             :     discproxy and proxy. The former contains data and functionality common to
      59             :     all discretizations, and this can be considered as an equivalent to a base
      60             :     class in the OOP sense. The latter, proxy, contains data and functionality
      61             :     specific to a particular discretization. When instantiated, Scheme is
      62             :     configured for a single specific discretization which must be selected from
      63             :     the list of types in SchemeBase::Proxy.
      64             : 
      65             :     The underlying type of proxy is a variant, which allows storing exactly one
      66             :     object. A variant is a type-safe union. An instance of a variant at any
      67             :     given time either holds a value of one of its alternative types. Read more
      68             :     on std::variant or boost::variant on how they work.
      69             : 
      70             :     All new member functions that comprise of the concept of the underlying
      71             :     proxies, i.e., the interface, must be defined in Scheme. Whereas common
      72             :     data, functionality, as well as the list of the proxy types that can be
      73             :     configured are defined in SchemeBase. Adding a new forwarding function
      74             :     either as a broadcast or addressing a particular chare array element can be
      75             :     done by simply copying an existing (similar) one and modifying what
      76             :     underlying function (entry method) it calls. The ones that forward to
      77             :     discproxy are grouped first, while the ones that forward to the specific
      78             :     proxy are listed as second. Using SFINAE, multiple overloads are (and can
      79             :     be) defined for a single function, depending on (1) whether it is a
      80             :     broadcast or addressing an array element, (2) whether it takes an optional
      81             :     (default) last argument, usually used for passing a CkEntryOptions object.
      82             :     You can see the Charm++-generated .decl.h file to see what (if any) default
      83             :     arguments a particular entry method may take.
      84             : 
      85             :     Currently, forwarding functions are defined for two types entry method
      86             :     calls: broadcasts, i.e., proxy.fn(), and addressing a particular element,
      87             :     i.e., proxy[x].fn(). Another third might be useful to add in the future and
      88             :     that is addressing an entry method behind a section proxy. As we have not
      89             :     used section proxies so far, this is not yet implemented, but if necessary,
      90             :     it should be relatively straightforward to do.
      91             : 
      92             :     Extending this class to other discretization schemes is done entirely in
      93             :     SchemeBase. Adding a new discretization scheme amounts to, at the minimum:
      94             :     (1) Adding a new type of Charm++ chare array proxy to SchemeBase::Proxy,
      95             :     (2) Adding a new type of Charm++ chare array element proxy to
      96             :         SchemeBase::ProxyElem, and
      97             :     (3) Adding a new branch to the if test in SchemeBase's constructor.
      98             : 
      99             :     Implementation details: All forwarding calls are implemented taking a
     100             :     variadic parameter pack, which can take any number of arguments (including
     101             :     zero) and use perfect forwarding to pass the arguments to the entry method
     102             :     calls. This way the code remains generic, easy to modify, and the compiler
     103             :     automatically adjusts the generated forwarding calls if the types and/or
     104             :     number of the arguments to any of the entry methods change. One exception to
     105             :     this is those forwarding calls that deal with default arguments, allowing
     106             :     for passing CkEntryOptions objects. There the number of arguments are
     107             :     hard-coded in the SFINAE construct, but should also be straightforward to
     108             :     modify if necessary.
     109             : 
     110             :     The functors named as call_* are used to dispatch (at compile time) entry
     111             :     method calls behind proxy, whose type is different depending on what
     112             :     specific discretization type is configured in the constructor. All common
     113             :     functionality in the call_* functors are lifted over to SchemeBase::Call.
     114             :     This helps keeping the function-call-specific code in Scheme minimal and
     115             :     reuses the generic part in SchemeBase.
     116             : 
     117             :     Note that another way of doing the dispatch, that is now done using the
     118             :     call_* functors, could have been implemented using a (compile-, or runtime)
     119             :     associative container storing std::function objects which would store
     120             :     pre-bound function arguments. That would work, but there are three problems
     121             :     with such an approach: (1) std::function is not obvious how to pup, i.e.,
     122             :     send across the network, (2) std::bind cannot currently be used to bind a
     123             :     variadic number arguments and thus the bind calls would not be very generic,
     124             :     and (3) a runtime associative container would take additional state. (Note
     125             :     that problem (2) above could probably be solved with variadic lambdas, but
     126             :     the (1) and (3) remain.)
     127             : 
     128             :   \see Talk on [Concept-based runtime polymorphism with Charm++ chare arrays
