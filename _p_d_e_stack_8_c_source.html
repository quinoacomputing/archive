38 // Constructor: register all partial differential equations into factory
39 //! \details This constructor consists of several blocks, each registering a
40 //! potentially large number of entries in a partial differential equation
41 //! factory, a standard associative container. At this time, each type of
42 //! partial differential equation can be configured to use a unique _physics
43 //! policy_ and a unique _problem policy_. (More types of policies might be
44 //! introduced in the future.) Policy classes are template arguments to the
45 //! partial differential equation classes and influence their behavior in a
46 //! different way, abstracting away certain functions, e.g., how to set
47 //! problem-specific initial and/or boundary conditions and how to update
48 //! their coefficients during time integration. For more information on
49 //! policy-based design, see http://en.wikipedia.org/wiki/Policy-based_design.
50 //! This abstraction allows [separation of concerns]
51 //! (http://en.wikipedia.org/wiki/Separation_of_concerns).
52 //!
53 //! Since the functionality of the policies are orthogonal to each other,
54 //! i.e., they do not depend on each other or their host (the partial
55 //! differential equation class), a Cartesian product of combinations are
56 //! possible, depending on which policies are selected. _This constructor
57 //! registers all possible combinations of policies for all available
58 //! differential equations._ By _register_, we mean, an entry is recorded in
59 //! an associative container, a std::map, that associates a lightweight key of
60 //! type inciter::ctr::PDEKey, consisting of only an enum for each policy
61 //! type, to an std::function object that holds the constructor bound to its
62 //! arguments corresponding to a particular partial differential equation +
63 //! policies combination. Note that registering these entries in the map does
64 //! not invoke the constructors. The mapped value simply stores how the
65 //! constructors should be invoked at a later time. At some point later,
66 //! based on user input, we then instantiate only the partial differential
67 //! equations (and only those configurations) that are requested by the user.
68 //!
69 //! Since all partial differential equation types (registered in the factory)
70 //! "inherit" from a common "base", client-code is unform and generic, and
71 //! thus immune to changes in the inner workings of the particular partial
72 //! differential equations as long as they fullfill certain concepts, i.e.,
73 //! implement certain member functinos, enforced by the _common base_, PDE.
74 //! The words "inherit and "base" are quoted here, because the common base
75 //! does not use inheritance in the normal OOP sense and does not use
76 //! reference semantics, i.e., pointers, visible to client-code either. The
77 //! relationship is more of a _models a_-type, which simplifies client-code
78 //! and allows for the benfits of runtime inheritance with value-semantics
79 //! which is less error prone and easier to read. See more about the
80 //! _models-a_ relationship and its implementation in, e.g., PDE/CGPDE.h.
81 //!
82 //! The design discussed above allows the registration, instantiation, and
83 //! use of the partial differential equations to be generic, which eliminates
84 //! a lot of boiler-plate code and makes client-code uniform.
85 //!
86 //! _Details of registration using mpl::cartesian_product:_
87 //!
88 //! The template argument to mpl::cartesian_product requires a sequence of
89 //! sequences of types. We use vector of vectors of types, listing all
90 //! possible policies. The constructor argument to mpl::cartesian_product is a
91 //! functor that is to be applied to all combinations. mpl::cartesian_product
92 //! will then create all possible combinations of these types and call the
93 //! user-supplied functor with each type of the created sequence as a template
94 //! parameter. The user-supplied functor here is registerPDE, which, i.e.,
95 //! its constructor call, needs a single template argument, a class templated
96 //! on policy classes. This is the partial differential equation class to be
97 //! configured by selecting policies and to be registered. The arguments to
98 //! registerPDE's constructor are the factory, the enum denoting the
99 //! differential equation type, and a reference to a variable of type
100 //! std::set< ctr::PDEType >, which is only used internally to PDEStack
101 //! for counting up the number of unique differential equation types
102 //! registered, used for diagnostics purposes.
103 // *****************************************************************************

144 std::vector< inciter::CGPDE >
145 PDEStack::selectedCG() const
146 // *****************************************************************************
147 // Instantiate all selected PDEs using continuous Galerkin discretization
148 //! \return std::vector of instantiated partial differential equation objects
149 // *****************************************************************************
150 {
151  std::map< ctr::PDEType, ncomp_t > cnt; // count PDEs per type
152  std::vector< CGPDE > pdes; // will store instantiated PDEs
153 
154  auto sch = g_inputdeck.get< tag::discr, tag::scheme >();
155  if (sch == ctr::SchemeType::MatCG || sch == ctr::SchemeType::DiagCG) {
156 
157  for (const auto& d : g_inputdeck.get< tag::selected, tag::pde >()) {
158  if (d == ctr::PDEType::TRANSPORT)
159  pdes.push_back( createCG< tag::transport >( d, cnt ) );
160  else if (d == ctr::PDEType::COMPFLOW)
161  pdes.push_back( createCG< tag::compflow >( d, cnt ) );
162  else Throw( "Can't find selected CGPDE" );
163  }
164 
165  }
166 
167  return pdes;
168 }
