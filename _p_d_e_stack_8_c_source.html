25 // Constructor: register all partial differential equations into factory
26 //! \details This constructor consists of several blocks, each registering a
27 //! potentially large number of entries in a partial differential equation
28 //! factory, a standard associative container. At this time, each type of
29 //! partial differential equation can be configured to use a unique _physics
30 //! policy_ and a unique _problem policy_. (More types of policies might be
31 //! introduced in the future.) Policy classes are template arguments to the
32 //! partial differential equation classes and influence their behavior in a
33 //! different way, abstracting away certain functions, e.g., how to set
34 //! problem-specific initial and/or boundary conditions and how to update
35 //! their coefficients during time integration. For more information on
36 //! policy-based design, see http://en.wikipedia.org/wiki/Policy-based_design.
37 //! This abstraction allows [separation of concerns]
38 //! (http://en.wikipedia.org/wiki/Separation_of_concerns).
39 //!
40 //! Since the functionality of the policies are orthogonal to each other,
41 //! i.e., they do not depend on each other or their host (the partial
42 //! differential equation class), a Cartesian product of combinations are
43 //! possible, depending on which policies are selected. _This constructor
44 //! registers all possible combinations of policies for all available
45 //! differential equations._ By _register_, we mean, an entry is recorded in
46 //! an associative container, a std::map, that associates a lightweight key of
47 //! type inciter::ctr::PDEKey, consisting of only an enum for each policy
48 //! type, to an std::function object that holds the constructor bound to its
49 //! arguments corresponding to a particular partial differential equation +
50 //! policies combination. Note that registering these entries in the map does
51 //! not invoke the constructors. The mapped value simply stores how the
52 //! constructors should be invoked at a later time. At some point later,
53 //! based on user input, we then instantiate only the partial differential
54 //! equations (and only those configurations) that are requested by the user.
55 //!
56 //! Since all partial differential equation types (registered in the factory)
57 //! "inherit" from a common "base", client-code is unform and generic, and
58 //! thus immune to changes in the inner workings of the particular partial
59 //! differential equations as long as they fullfill certain concepts, i.e.,
60 //! implement certain member functinos, enforced by the _common base_, PDE.
61 //! The words "inherit and "base" are quoted here, because the common base
62 //! does not use inheritance in the normal OOP sense and does not use
63 //! reference semantics, i.e., pointers, visible to client-code either. The
64 //! relationship is more of a _models a_-type, which simplifies client-code
65 //! and allows for the benfits of runtime inheritance with value-semantics
66 //! which is less error prone and easier to read. See more about the
67 //! _models-a_ relationship and its implementation in, e.g., PDE/CGPDE.h.
68 //!
69 //! The design discussed above allows the registration, instantiation, and
70 //! use of the partial differential equations to be generic, which eliminates
71 //! a lot of boiler-plate code and makes client-code uniform.
72 //!
73 //! _Details of registration using brigand::for_each and
74 //! tk::cartesian_product:_
75 //!
76 //! The template argument to brigand::for_each, as used below, requires a
77 //! list of list of types. We use brigand::list of brigand::list of types,
78 //! listing all possible policies, where the inner list must have exactly two
79 //! types, as the list of lists is constructed from two lists using the
80 //! cartesian product, and the length of the outer list (the list of lists) is
81 //! arbitrary. The constructor argument to brigand::for_each is a functor that
82 //! is to be applied to all members of the outer list. tk::cartesian_product
83 //! will create all possible combinations of these types and call the functor
84 //! with each type of the created sequence as a template parameter. The
85 //! functor here inherits from registerPDE, which, i.e., its constructor call,
86 //! needs a single template argument, a class templated on policy classes.
87 //! This is the partial differential equation class to be configured by
88 //! selecting policies and to be registered. The arguments to registerPDE's
89 //! constructor are the factory, the enum denoting the differential equation
90 //! type, and a reference to a variable of type std::set< ctr::PDEType >,
91 //! which is only used internally to PDEStack for counting up the number of
92 //! unique differential equation types registered, used for diagnostics
93 //! purposes.
94 // *****************************************************************************
95 {
96  registerTransport( m_cgfactory, m_dgfactory, m_eqTypes );
97  registerCompFlow( m_cgfactory, m_dgfactory, m_eqTypes );
98 }

101 PDEStack::selectedCG() const
102 // *****************************************************************************
103 // Instantiate all selected PDEs using continuous Galerkin discretization
104 //! \return std::vector of instantiated partial differential equation objects
105 // *****************************************************************************
106 {
107  std::map< ctr::PDEType, ncomp_t > cnt; // count PDEs per type
108  std::vector< CGPDE > pdes; // will store instantiated PDEs
109 
110  auto sch = g_inputdeck.get< tag::discr, tag::scheme >();
111  if (sch == ctr::SchemeType::MatCG || sch == ctr::SchemeType::DiagCG) {
112 
113  for (const auto& d : g_inputdeck.get< tag::selected, tag::pde >()) {
114  if (d == ctr::PDEType::TRANSPORT)
115  pdes.push_back( createCG< tag::transport >( d, cnt ) );
116  else if (d == ctr::PDEType::COMPFLOW)
117  pdes.push_back( createCG< tag::compflow >( d, cnt ) );
118  else Throw( "Can't find selected CGPDE" );
119  }
120 
121  }
122 
123  return pdes;
124 }
